Notes and considerations

Regarding the async attribute
The current W3 validation spec does not allow an attribute value on async to validate. So, something like <script src="ux/file.js" async="true"></script> would automatically be reevaluated to <script src="ux/file.js" async=""/> because it is empty, I think that converting scripts to the lazy loader and getting the relative file path correct automatically would be preferred, ie <script>app.include('../../ux/file.js', false);</script>
Processing Instructions
Though this thouroughly breaks application template principles, I'm interested to see if it's possible to evaluate php processing instructions as a proof of concept. This would be done with something like $xpath->evaluate('string(//processing-instruction("php")[1])');. There may be a simple use case for allowing some limited config to partial templates. To be clear, this would not be adding source code, but processing instruction flags and such.
Parsing Nodes for MarkDown
I think this would be beneficial for cumbersome elements, like <time>, that could benefit from application-wide formatting, ie. +August 5, 2018+ could be parsed and output into configurable format. Another use case would be a custom tagging like in sound opinions, where {artist: Warpaint} was used to automatically provide additional functionality.. same principle here, whether it be auto-drawing links or constructing a data-store.
